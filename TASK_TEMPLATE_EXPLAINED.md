## 📚 Part 1: What is `build_output_file()`?

### The Two Types of Files

In this system, there are **TWO types of markdown files**:

### 1. **INPUT Task File** (Created by Watchers)
```
vault/Needs_Action/email_20260325_143022_Meeting.md
```

**Purpose:** Tell Claude what to do

**Created by:**
- `watchers/filesystem_watcher.py` → `create_file_drop_task()`
- `watchers/gmail_watcher_oauth.py` → `create_email_task()`
- `watchers/gmail_watcher_imap.py` → `create_action_file()` (CUSTOM)

**Content:**
```markdown
---
type: email
task_id: email_20260325_143022_Meeting
from: boss@company.com
subject: Meeting Tomorrow
received: 2026-03-25T14:30:22
priority: high
status: pending
---

# Email: Meeting Tomorrow

**From:** boss@company.com
**Priority:** High

---

## Body

[Email content here...]

---

*Generated by AI Employee Gmail Watcher*
```

**Characteristics:**
- ✅ Has `status: pending`
- ✅ Contains original content (email body, file content)
- ✅ Asks Claude to make a decision
- ✅ Goes INTO Claude for processing

---

### 2. **OUTPUT Result File** (Created by Claude Runner)
```
vault/Done/RESULT_email_20260325_143022_Meeting.md
```

**Purpose:** Record what Claude decided and did

**Created by:**
- `claude_runner.py` → `build_output_file()`

**Content:**
```markdown
---
type: email_result
task_id: email_20260325_143022_Meeting
original_name: Meeting Tomorrow
source_file: ""
original_task: "[[Processing_Archive/email_20260325_143022_Meeting.md]]"
status: completed
processed_at: 2026-03-25T14:35:00
ai_decision: complete_task
ai_category: email
---

# Meeting Tomorrow

**Processed:** 2026-03-25 14:35:00
**Decision:** complete_task
**Category:** Email

---

## Summary

Claude's summary of what it did

---

## AI response

Claude's detailed response

---

## Action taken

What action Claude performed
```

**Characteristics:**
- ✅ Has `status: completed` (or `pending_approval`, `needs_revision`)
- ✅ Contains Claude's decision and response
- ✅ Records what happened
- ✅ Comes OUT of Claude processing

---

## 🎯 Part 2: What Does `build_output_file()` Actually Do?

### Function Signature
```python
def build_output_file(
    task_id: str,              # "email_20260325_143022_Meeting"
    task_type: str,            # "email" or "file_drop" or "whatsapp"
    original_name: str,        # "Meeting Tomorrow"
    original_path_obsidian: str,  # "" (empty for email)
    decision: dict,            # Claude's JSON response
    processed_at: datetime,    # When Claude finished
) -> str:                      # Returns complete markdown string
```

### What It Builds

```python
output_md = f"""---
type: {task_type}_result                    # ← "email_result"
task_id: {task_id}                          # ← From parameter
original_name: {original_name}              # ← From parameter
source_file: "{original_path_obsidian}"     # ← From parameter
original_task: "[[Processing_Archive/{task_id}.md]]"
status: {status}                            # ← From decision
processed_at: {processed_iso}               # ← From parameter
ai_decision: {decision.get("decision")}     # ← From Claude's JSON
ai_category: {decision.get("category")}     # ← From Claude's JSON
---

# {original_name}

**Processed:** {processed_str}
**Decision:** {decision.get("decision")}
**Category:** {decision.get("category").title()}

---

## Summary

{decision.get("summary", "No summary provided")}

---

## AI response

{decision.get("response", "No response provided")}

---

## Action taken

{decision.get("action_taken", "No action recorded")}
"""
```

### Example Input → Output

**Input (Claude's JSON):**

## 📊 Part 3: Should We Centralize Templates?

### Current State

| Component | Template Used | Central or Custom? |
|-----------|--------------|-------------------|
| File System Watcher | `create_file_drop_task()` | ✅ Central |
| Gmail OAuth Watcher | `create_email_task()` | ✅ Central |
| Gmail IMAP Watcher | `create_action_file()` | ❌ CUSTOM |
| Claude Runner (output) | `build_output_file()` | ✅ Central |

### The Question

**Should Gmail IMAP Watcher use central template?**

---

## Option A: Keep Custom Template (Current)

**Gmail IMAP has its own `create_action_file()` method:**

```python

    # Custom template with ALL Gmail-specific fields
    task_content = f"""---

gmail_link: {gmail_link}                # ← Gmail-specific

# AI Processing Status                    # ← Gmail-specific
ai_status: pending
ai_processed_at: [PENDING]
ai_decision: [PENDING]
ai_category: [PENDING]
ai_summary: [PENDING]
---

## Email Information (Table)              # ← Gmail-specific
| Property | Value |
|----------|-------|
| Thread   | Yes   |

## Instructions for AI                   # ← Gmail-specific
1. Identify email type...
2. Check if this is part of a thread...
...

```

**Pros:**
- ✅ All Gmail features included
- ✅ Complete control over format
- ✅ Can add fields anytime without affecting others

**Cons:**
- ❌ Duplicate code (template logic in 2 places)
- ❌ Inconsistent with other watchers
- ❌ More maintenance (2 templates to update)
- ❌ Future watchers need to choose: central or custom?

**Future Impact:**
- If you add WhatsApp watcher → Need custom template?
- If you add SMS watcher → Need custom template?
- Every new watcher = another custom template?

## Option B: Use Central Template Only

**Gmail IMAP calls `create_email_task()` from central:**

```python
from utils.task_template import create_email_task

task_id, task_content = create_email_task(
    from_address=headers['from'],

    content=body,
    timestamp=received_date,

```

**Pros:**
- ✅ Single source of truth
- ✅ Consistent across all watchers
- ✅ Easy maintenance (one place to update)

**Cons:**
- ❌ **LOSES ALL GMAIL-SPECIFIC FIELDS:**
  - No `filter_reason` (why was email selected?)
  - No `is_reply` (thread detection)
  - No `gmail_message_id` (permanent ID)
  - No `gmail_link` (clickable link)
  - No AI Processing Status section
  - No Email Information table
  - No Instructions for AI section

**Verdict:** ❌ **NOT ACCEPTABLE** - Loses critical Gmail features

## Option C: Enhanced Central Template (RECOMMENDED)

**Add new function to `task_template.py`:**

```python
# utils/task_template.py

def create_email_task_enhanced(
    from_address: str,
    to_address: str,
    subject: str,
    content: str,
    timestamp: datetime,
    priority: str,
    filter_reason: str = "",
    is_reply: bool = False,
    gmail_message_id: str = "",
    gmail_link: str = "",
) -> Tuple[str, str]:
    """
    Enhanced email task template with Gmail-specific metadata.
    
    Includes:
    - filter_reason: Why this email was selected
    - is_reply: Thread detection
    - gmail_message_id: Permanent Gmail ID
    - gmail_link: Direct link to email
    """
    task_content = f"""---
type: email
task_id: {task_id}
from: "{from_address}"
to: "{to_address}"
subject: "{subject}"
received: {timestamp.isoformat()}
priority: {priority}
status: pending
retry_count: 0
filter_reason: {filter_reason}
is_reply: {str(is_reply).lower()}
gmail_message_id: {gmail_message_id}
gmail_link: {gmail_link}
---

# Email: {subject}

**From:** {from_address}
**To:** {to_address}
**Received:** {timestamp.strftime('%Y-%m-%d %H:%M:%S')}
**Priority:** {priority.title()}
**Filter Reason:** {filter_reason}
**Thread:** {"Yes (reply)" if is_reply else "No (original)"}

---

## Body

{_truncate_content(content)}

---

*Generated by AI Employee Gmail Watcher*
*Task ID: `{task_id}`*
*📧 [Open in Gmail]({gmail_link})*
"""

    return task_id, task_content
```

**Gmail IMAP uses it like this:**

```python
from utils.task_template import create_email_task_enhanced

# Get base template
task_id, task_content = create_email_task_enhanced(
    from_address=headers['from'],

    content=body,

    priority=priority,
    filter_reason=filter_reason,

)

# Append Gmail-specific sections
task_content += f"""

## AI Processing Status

ai_status: pending
ai_processed_at: [PENDING]
ai_decision: [PENDING]
ai_category: [PENDING]
ai_summary: [PENDING]

---

## Instructions for AI

1. **Identify the email type** (request, invoice, informational, reply, etc.)
2. **Check if this is part of a thread** (see "Thread" info above)
3. **Classify urgency and category** based on content
4. **Apply business rules** from Business_Goals.md and Company_Handbook.md
5. **Return JSON decision** as defined in CLAUDE.md
"""
```

**Pros:**
- ✅ Central location (single source of truth)
- ✅ All Gmail features preserved
- ✅ Consistent across watchers
- ✅ Easy maintenance
- ✅ Future watchers can use appropriate template
- ✅ Clear separation: common fields vs. specific fields

**Cons:**
- ⚠️ Requires updating `task_template.py`
- ⚠️ Gmail IMAP still needs to append custom sections (but less code)

**Verdict:** ✅ **BEST** - Combines centralization with flexibility
